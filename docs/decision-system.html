How an Agent Decides · Hearsay / Market Town

Every NPC acts on what it believes, not on what is true. This page walks through the decision pipeline that runs each cognition tick — from raw perception to a footstep — and lets you watch one agent's mind change, field by field, as a threat walks into view.

01The epistemic split

The single load-bearing invariant. Decisions read beliefs only; execution reads ground truth. An agent can be genuinely fooled — by a disguise, a planted rumour, a scarecrow dressed as a person — while the physics of the world still resolve correctly.

Ground truth · execution

Real positions, factions, damage, who is actually hostile. Read by combat, movement, and isHostile — never to decide, only to resolve.

combat.js · movement · Simulation.isHostile · onCombatEvents
cognition boundary

Beliefs · cognition

A private, per-agent, decaying N×M table — for each of N observers, a row of beliefs about the M topics that agent tracks. A topic is whatever it can hold an opinion about, not only other people. Everything an agent does to decide reads here.

beliefs.js (BeliefStore) · perceive · gossip · reason · decide
The split is a cognition boundary, not an observer boundary. The Director, Chronicle and Gazette read the whole roster's truth to narrate world history — that's fine. The test is: “does this read drive an agent's behaviour?” If yes, it must go through beliefs.

What is M? Not just “every other agent.” A belief row is keyed by a topic id — and a topic is anything the agent can hold a confidence-weighted, decaying, provenance-tagged opinion about. The store is sparse: M is the handful of topics this agent actually tracks, not the whole world.

So the same record type spans kinds. A place is a percept with a namespaced id (B:<n>, disjoint from agent ids) carrying placeKind, sheltered, benefitFelt; a believed motive rides on an agent topic as destId/intent (“headed to the gate to flee”); an item / recipe belief lives in the Know(topic) model with its own value & provenance; an association (assoc) links an agent topic to a place — “I think Korg stashes loot there.”

04bIs a belief a struct or a proposition?

The card above shows the as-built record — and it's worth being honest that it's a denormalized projection, not the general thing. A belief wants to be a proposition: something with a value, a confidence, a source, and an age. The flat struct hard-codes which propositions are believable as columns — fast, but lossy.

As-built · denormalized row

one fixed struct per (observer, topic)

Believable = whatever has a column. Confidence / provenance / recency are mostly whole-record scalars. Same struct for every topic kind, so most fields sit null.

+ fastflat, cache-friendly, vectorizable
+ cheap decayone scalar per record
– closed ontologyno column ⇒ can't believe it
– coarsecan't be sure of faction, unsure of position

General · a set of facts

{ topic, attribute, value, confidence, provenance, observedAt }

A belief is a proposition. The struct above is just these facts grouped by topic and flattened. Per-fact confidence, per-fact source, open attribute set, naturally sparse.

+ openbelieve anything, no schema edit
+ preciseconfidence/age per fact, not per record
– costa fact map per agent; slower to scan
≈ foreshadowedKnow(topic) already works this way
The lossiness, concretely. Belief: “Korg is a raider, last seen at the gate heading there to flee — and he owes me 5 gold.” The struct stores lastFaction/hostile/lastPos/destId/intent — but “owes me 5 gold” has no column, so it's lost or stuffed into the untyped knownDeeds[] bag. As facts it's just five more rows: (Korg, faction, raider, c=.7, rumor) · (Korg, hostile, true, c=.6, rumor) · (Korg, at, gate, c=.9, seen) · (Korg, intent, flee, c=.5, inferred) · (Korg, owes_me, 5, c=1.0, ledger). The as-built trades that generality for tick-rate speed across thousands of agents — a deliberate bet that the propositions that matter are bounded and known in advance.

05Inside decide: goals & GOAP

decide isn't a switch statement — it's three steps over a data-driven grammar.

① deriveGoals

motivation.js · DERIVERS registry

Each deriver reads the agent's own beliefs / needs / episodic memory and may push a goal onto the stack. goalAvenge from an assaulted memory, goalRepay from a succoured one, goalSeekFortune from a wealth ambition. Adding a motive = registering a row.

② pruneGoals + score

motivation.js · decide.js

Expired / satisfied goals are popped (awarding closure XP). Then a utility argmax over fight · flee · eat · work · rest · socialize · market · wander — tilted by ambition & quirk, with Schmitt-band hysteresis so agents don't dither, and a survival-before-commerce gate.

③ plan (GOAP)

planner.js · backward-chaining

For a goal that needs steps (“have 10 gold”), the planner backward-chains over PRIMITIVES: find a verb whose effect unifies the open Atom, add its preconditions as new subgoals, recurse (depth ≤ 5), keep the min-cost feasible chain (≤ 8 steps).

Verbs are data, not control flow. The verb→executor binding lives in a registry (exec/registry.js): EXECUTORS (verb → world-interaction on arrival), DERIVERS (belief → goal), EFFECT_HOLDS (did the step land?), and PLAN_OUTCOME (a watched act resolved → caution). Adding a behaviour means registering a row, never editing a switch. Each feature is one file in js/sim/features/.

06From goal to footstep

act(dt) runs every frame. Locomotion is one potential-field primitive: the goal's kind indexes STEER_FILLS, which builds a Field of attractors and repulsors from the agent's own beliefs; steer() sums them into a heading. On arrival it fires the explicit world-interaction verb — gather, strike, produce, build, transfer.

flee

STEER_FILLS['flee']

pure repulsor at the threat's believed lastPos — pushes away, never “arrives”.

work / market / comfort

STEER_FILLS[kind]

attractor at the best believed POI; on arrival → produce(), a trade, or restore comfort + stamp the felt benefit onto the place-belief.

flee + safe / hide

STEER_FILLS['avoid']

threat repulsor blended with a safe/conceal attractor from the static mental map.
